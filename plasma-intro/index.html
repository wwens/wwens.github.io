Plasma 从入门到入坑

1 Plasma concept

使用可行的Challange/Response计算模型, 支持多方可扩展的链下计算

Plasma是一个带激励的强制智能合约执行框架。这些智能合约

  • 通过网络手续费用于激励持续的自动化运作
  • 最终依赖于低层的区块链来强制交易状态的锁定
  • 使用欺诈证明可以强制主链上状态的锁定，周期性的将正常状态的Merkle证明广播到主链。

Plasma由五个核心部分构成：

  1. 一个激励层，用于持久的以优化的价格执行合约，以一种树状态形式来组织子链来尽可能的提高成本效率和网络交易清算的效率。
  2. 一个MapReduce框架，构建一个状态转换的欺诈证明，在嵌套的子链中，兼容树结构同时重组状态转换为可扩展。
  3. 一个共识机制，依赖于主链，尝试复制Nakamoto共识激励。
  4. 一个位图的UTXO提交结构，保证在主链下的确定的状态转换，共识尽可能降低退出费用。
  5. 允许在数据不可用或者其他Byzantine行为时可以退出，也是Plasma的运行中的关键设计点。

2.3 Sparse Merkle Tree

稀疏默克尔树 SMT 跟 标准默克尔树 MT很想，只是它包含的数据是序列化有索引的，并且每个数据点都放在与该数据点索引对应的叶子上。

  • 数据结构MT的优势在于当判断一个数据包含在MT中时可以只需要相关的O(log(n))的数据即可，但在判断一个数据不包含在MT中时却并没有优势，仍需要全部的数据。
  • 数据结构SMT的优势在于判断一个数据不包含在SMT中时，仍可以优化为需要相关的O(log(n))的数据。

2.4 Transaction

一个交易包含以下几个部分： 

[[prev_hash, prev_block, (target_block?), token_id, new_owner], signature]

uid: it’s unique position in the SMT (a leaf).

previousBlock: every transaction must point to the previous block where the previous transaction occurred in. This creates a chain of history. If it’s a deposit, it is “0”.

denomination: This how much the the specific token holds. These denominations can be anything. If it’s used for an ERC20 fungible (made non-fungible), you essentially create a denomination based on the deposit that’s similar to a note in real life. Eg, a 20 GNO non-fungible token or a 13 GNO non-fungible token.

new_owner The new owner of the transaction.

signature: Signature on the transaction’s hash.

hash: The hash of the transaction’s details (unsigned).

merkle_hash: The hash of transaction’s details that’s also signed by the sender that’s to be included in the merkle tree.

sender: The person transferring the token.

2.5 Exits

针对全局强制非全局数据的数据可用性问题：

  • 区块扣留攻击
  • Plasma的核心是针对有问题链的退出机制设计

应对数据可用性问题的退出机制 :

2.5.1 Spent Tx Challenge:

退出交易tx试图退出已花费的不合法交易， Spent Tx Challenge 将立即使退出无效并且要求奖励。

2.5.2 Double Spend Challenge:

退出交易tx试图退出包含双重的交易， Double Spend Challenge 将立即使退出无效并且要求奖励。

2.5.3 Invalid History Challenge:

欺诈性的运营者可以选择包括从未附加到原始存款的点开始的交易，基本上从空气中创建历史。。如果运营者故意扣留块，也会发生此无效历史记录。 Invalid History Challenge 需要交互， 就是说一旦受到挑战，另一个用户需要提交跟随它的交易，以证明历史确实有效。为了提出质疑，还需要提交保证金，如果提供后续交易，则会减少保证金。如果没有后续交易，那么它意味着历史确实有效。

3 Problems & Improvement

3.1 Data Availiability

数据可用性问题可以认为是当前区块链领域遇到的最大最难的问题之一，根源在于扩展区块链的同时我们的计算模型发生了变化。

  • 原来：区块链上的数据 -> 计算，改动 -> 区块链上的数据
  • 扩展后：区块链上的数据 + 外部数据 -> 计算，改动 -> 区块链上的数据

这就意味着在保证区块链上数据正确的同时还需要保证整个区块链p2p网络中的数据有一定的可用性。这个问题当下业内的思路是概率优化，做到完全解决很难，我们考虑引入欺诈证明、零知识证明、机制设计等方案优化数据不可用的问题。

3.2 Light client & Fraud proofs

区块链引入轻客户端做数据验证有很大的诱惑，可以做到O(log(n))的网络验证效率，在未来的区块链应对可扩展性和分片、跨链等领域都有着不可忽略的优势。同时由于轻客户端存在数据不可用的问题，需要区块链网络有反欺诈证明的能力，我们考虑通过一种全节点提供数据缺陷保险服务和充实区块审计服务的方案应对欺诈证明问题。 引入零知识证明的优势： 应对机密交易和隐私问题，零知识证明让我们在传统密码学保障的安全性的技术上可以做到高度机密需要隐私保护的交易数据隐藏。 不仅如此，零知识证明还可以优化区块链计算过程中的数据量大小和计算量大小，这对于区块链网络中需要大量数据交互和验证的场景来说意义重大。
