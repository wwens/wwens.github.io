1 Hashgraph算法在公网环境下的问题

Hashgraph共识算法通过Gossip网络和虚拟投票策略达成交易顺序的共识，该共识的前提是要求网络节点超过2n/3的投票能力具有对famouswitness事件的一致投票结果，其中n是全网的当前投票能力总和，该投票能力通常为节点的持股数量。由于采用了本地投票策略，Hashgraph可以实现较快的交易确认速度。然而该方法存在以下问题：

  1. 在广域网环境中，节点波动性较强，全网的投票能力n的波动也随之增强，这可能导致系统长时间无法找到满足2n/3投票一致的事件，从而无法达成共识。
  2. 受节点稳定性、处理能力、带宽等因素影响，不同节点处理事件的能力差别较大。若系统中存在大量能力较弱的节点参与投票，同样会造成系统长时间无法达成共识。
  3. 广域网环境下，节点频繁波动可能导致节点被分割成多个子网。根据Gossip邻居交换协议，节点会周期性剔除长时间未更新的邻居。当邻居稳定后，节点可在子网内达成共识。此时若子网规模较小，很容易使恶意节点在同一轮产生两个以上famouswitness事件，从而产生双花交易。
  4. 随着系统规模增大，节点收到的同步信息越来越多，可以预见系统的吞吐率会随节点数目的增加而降低。

2.1 PlanA：带权限系统的维护白名单一致合约

核心思想: 采取一定程度的中心化控制，主要思想为通过带有权限控制的合约类型交易，全网节点达成一致的白名单节点数共识。

个人认为存在的问题：

  1. 认为该问题本质为网络层的问题，要解决的问题是如何让在新加入节点的情况下，让全网在白名单节点数上要有基本的一致，准确的说 对大多数的理解一致 。
  2. 认为该问题和账本共识逻辑无关， 不存在自动达成共识的可能性 ，或者 要么区块链就是在解决不存在的问题要么必然引发n多其他问题 。
  3. 问题体现在以下几点：
    • 每个节点在监测到本地白名单和p2p网络的差异后会发起受公共合约权限保护的合约交易
      • 每个节点都发，合约账户只允许1个单位Token存在的设计，本质上是过滤出一个一致。
      • 即便可以过滤出一致来，仍存在不知道何时应该更新合约权限范围内的白名单的问题，因为该过程仅能说明有的节点发送的交易余额够，能够正确执行，有的不够不能正确执行，并没有实现共识，p2p网络中节点的白名单的实际状况仍是未知不可控的。
      • 对上述 并没有实现共识 的进一步解释是，节点之间并没有相互的信息校验及确认过程存在，不知道和共识有什么关系。
  4. 综合之前多次的讨论大概是以下思路：
    • 假设网络已存在5个节点:A、B、C、D、E，新节点F要加入网络，各节点白名但即为Numn
      • (1) A说Numn=6，其余各节点都改为Numn=6
      • (2) 设计一些特殊的交易加入账本共识逻辑中
      • (3) 依赖一个已共识的交易中不该动的数据，即寄希望于智能合约来实现
    • 上述三种方案，无论怎样设计都是假设拜占庭将军问题不存在天然就是共识的，都是要在代码中写一段死逻辑，故并没有实现共识 ，且 放大了风险，并不是控制了风险 ，因为任何一个节点都存在 单点故障 ，有篡改数据的能力，并影响算法的正常逻辑。
  5. 个人理解的共识包括但不限于:
    • 共识信息在多个节点之间可以多次相互校验和确认，并存在相应过程。
    • 认为白名单的数量这个信息在当下的设计中不是一个可以校验的信息，个人看到三种可能的方向
      • 1> 从一个固定位置，即采用中心化的方案。
      • 2> 借鉴btc破解哈希问题的难度nonce的设计，即数据来自从历史演进的账本不可篡改的记录。
      • 3> 本质上是设计一个独立解决白名单一致的共识机制，传统分布式领域处理一致性的方案也是可以的。
  6. 基于上述原因，该过程和Hashgraph的共识并无任何关系。
  7. 支持该过程需要设计额外的交易结构和合约设计对项目未来迭代看不到明显益处。

2.3 PlanB：白名单同步及监控报警服务

核心思想: 采用中心化Server的白名单List来解决网络的白名单更新问题，强制所有节点必须定时拉取同步白名单的List。

个人认为存在的问题：

  1. 你说我中心化，我们就聊点别的先。
  2. 有 单点故障问题 ，且每个节点都有单点故障问题，故假设是所有节点都正常。
  3. 突破性的解决该问题，只能走 算法升级 的路，没有其他真正有意义的方案，可行都很难做到，共识算法对区块链项目的意义由此可见一斑。
  4. 便于管理、升级、维护和做一些统计的事情。

2.4 PlanB技术Spec

最简单的方式

  • 在一个专门的服务器上维护一个白名单List的文件
  • Node节点中加上定时同步拉取逻辑，并Merge节点本地白名单List
